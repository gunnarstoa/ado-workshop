Troubleshooting & Common Errors

Diagnose and resolve the most frequent ADO automation and production issues
Reference Use As Needed

This page covers the most common errors encountered when automating ADO with Workflow and when promoting ADO from Development to Production. Each section describes the symptom, root cause, and resolution steps.

Workflow Issues

Error: Workflow Fires but the Wrong Action Runs

Symptom

The Workflow schedule triggers on time, Run History shows "Completed," but the data in the target model has not been updated — or the wrong model received data.

Root Cause: The ADO Action step inside the Workflow template is pointing to the wrong link. This most commonly happens when a production template was duplicated from a development template but the step was not updated to reference the production link. The template still runs the Dev link.

Resolution:

  1. Open the Workflow template that ran.
  2. Check the ADO Action step — confirm which link it references.
  3. If it references the Dev link when you expected a Prod link: edit the step, change the link to the correct production link, and save.
  4. Run the template again manually using "Run Now" and verify the correct model receives data.
Prevention

After duplicating a Workflow template for production use, always open the new template and verify the ADO Action step before enabling the schedule. One wrong reference can silently run the wrong pipeline for days.

Error: Schedule Doesn't Trigger

Symptom

The expected run time passes, but the Workflow template shows no new entry in Run History. It appears the schedule never fired.

Common Causes and Fixes:

Cause How to Identify Fix
Schedule is disabled Template shows "Not scheduled" or Enabled toggle is OFF Open the template → Schedule tab → toggle Enabled to ON → save
Wrong timezone math — run time already passed for today Expected 11 PM local but set 23:00 without accounting for UTC offset Recalculate the UTC time, update the schedule time field
No action in the template Template has zero steps — ADO Action was never added or was deleted Add the ADO Action step and re-enable the schedule
Workspace quota exceeded Admin console shows workspace size warnings or errors Free up space in the workspace or request quota increase from your Anaplan admin

Productionization Issues

Error: Transformation View Still Points at Dev Data After Production Link Created

Symptom

The production pipeline runs and loads data into HR Model (Prod), but you notice the data matches Dev source data exactly — it looks like the same small sample set rather than production data. Alternatively, changes made to the production CSV file are not reflected in the model after a run.

Root Cause: After cloning a link using "Clone to LLM models," the production link's source remains set to the development transformation view. Production data is never actually read — the pipeline is still pulling from Dev source data.

Resolution:

  1. Open ADO → Links. Find the production link.
  2. Check the Source field. If it shows a Dev transformation view, that is the problem.
  3. Change the source to the production transformation view (the one you created by duplicating and updating the Dev view).
  4. Save the production link.
  5. Run the pipeline again manually and verify the correct data loads.
Root of the Problem

The "Clone to LLM models" feature clones the model target of the link (pointing it at the production model), but it does not clone the source. You must always update the source manually after cloning. This is the single most common mistake in ADO productionization.

Error: Source Data Not Connected Correctly — No Data Loaded

Symptom

The ADO pipeline runs and shows "Completed" status, but the target Anaplan model has no new data or the module remains empty after the run.

Common Causes and Fixes:

  1. Source data file is empty or has wrong headers: Open the source dataset in ADO and preview the data. If it shows 0 rows or the column headers don't match what the transformation view expects, fix the CSV file and re-upload.
  2. Transformation view filter removes all rows: Open the transformation view and run a preview. If the output shows 0 rows but the source shows rows, check your filter logic — it may be too restrictive for the production data.
  3. Link column mapping broken: Open the link and review the column-to-list/module mappings. If the production model has different list member names than expected, mappings may fail to resolve — resulting in 0 rows loaded. Verify that the production model's lists are populated with the correct members.
  4. Workspace size insufficient: The Anaplan workspace may not have enough available space to receive the data. Check Workspace Admin → Workspace Size. If the workspace is near capacity, free up space before re-running.

Error: Connection Failures in Production

Symptom

If using a live connector (not CSV), the ADO pipeline fails with a connection error. Run History shows "Failed" with an error message referencing authentication, network, or timeout.

Common Causes:

  • Wrong credentials: Production connection was configured with dev/sandbox credentials. Re-open the production connector configuration in ADO and update the username, password, or API key to production credentials.
  • IP allowlisting not configured: The source system's firewall is blocking connections from Anaplan's IP range. Provide Anaplan's IP addresses to the customer's IT team and request they be allowlisted.
  • TLS/SSL certificate mismatch: The production system uses a self-signed certificate that ADO doesn't trust. Work with the customer's IT team to install a trusted certificate or configure the connector to accept the certificate.
  • Network timeout — large dataset: The source system is slow to respond (large dataset extract takes more than the connector timeout). Optimize the query at the source — pre-aggregate or filter records before ADO pulls them. Refer to Planual rule 8.01-07 on connector performance factors.

Error: Transformation View Errors After Source Change

Symptom

After duplicating a transformation view and changing its source to production, the transformation steps show errors — red indicators on specific steps, or columns that were referenced in expressions are now missing.

Root Cause: The production source data has different column headers from the Dev source data. The transformation steps reference specific column names, and those references break when the column names change.

Resolution:

  1. Open the production source dataset and note the exact column header names.
  2. Compare to the Dev source dataset column headers.
  3. In the production transformation view, click through each step that shows an error. Update the column references to match the production column names.
  4. As a best practice: standardize column naming in all source data files (dev and prod should have identical headers). This makes promotion trivial.
Best Practice

When setting up ADO during development, agree on a column naming standard with the customer's source system team. If Dev and Prod CSVs always have the same column headers, transformation views can be duplicated with zero changes — a major time saver during productionization.

General ADO Best Practices for Production

  • Use saved views, not raw modules, as ADO sources. If sourcing data from Anaplan itself, always export from a saved view (never the raw module or list). Saved views let you filter and control exactly what data ADO reads. (Planual 5.04-09)
  • Remove duplicate rows before the link. Duplicate rows in source data slow the link and can cause unpredictable behavior in Anaplan lists. Add a deduplication step in your transformation view before the final output. (Planual 8.04-03)
  • Use code mapping for numbered lists. When loading into Anaplan numbered lists, map the code field, not the name. Name mapping treats code as an attribute, which can cause unexpected list behavior. (Planual 8.04-06)
  • Reduce data volume as early as possible. Apply filters and aggregations in the first transformation steps. Don't pass 10 million rows through four transform steps only to filter them at the end. (Planual 8.02-01)
