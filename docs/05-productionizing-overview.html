Productionizing ADO

The 6-step flow to promote Dev ADO to a production environment
Productionization ⏱ 30 min read

Why Productionizing ADO Is Different

When a developer promotes an Anaplan model from Dev to Production using Anaplan's Application Lifecycle Management (ALM), the model structure and formulas travel across environments cleanly. ADO is different — ADO does not have a full ALM promotion path. You cannot simply "promote" an ADO pipeline the way you promote a model revision.

Instead, you rebuild each ADO component for the target environment, using the development pipeline as your blueprint. The one exception is links, which have a built-in clone feature. Everything else — source data and transformation views — must be duplicated manually.

The 6-Step Production Promotion Flow

Starting Point

These steps assume you have: (1) a working Dev ADO pipeline with at least one link pushing data into a development Anaplan model, and (2) a need to create an equivalent pipeline that loads production data into a production Anaplan model. The development model and its ADO pipeline are your blueprint.

1

Prepare the Production Model

Before any ADO work, you need a production Anaplan model. Use Anaplan's standard model promotion process (ALM or manual export/import) to create the production model from the development model. The production model will have the same structure as Dev but with no data in it — data will flow in through the production ADO pipeline.

Note

If you're using Anaplan's standard ALM (Application Lifecycle Management), you should already have a model in your Linked Library (LLM) that you can sync to production. This step ensures the production model exists and is synced before you configure ADO against it.

2

Create Production Source Data

Your development pipeline points at development source data (a dev CSV file or a connection to a dev/sandbox system). For production, you need production source data. How you do this depends on your connection type:

If using CSV

In ADO, upload a production CSV file (one that contains production-representative data) and create a new source dataset pointing to it. Name it clearly: e.g., Employee Data — Production.

If using a Connector

Create a new connector configuration in ADO that points to the production system. Use production credentials, not dev/sandbox credentials. Name the connection clearly: e.g., HRIS — Production.

Warning

Never reuse a development connection or CSV for production. Dev data often contains test records, masked values, or sample rows that are not appropriate for a production pipeline. Always use production-sourced data in the production ADO pipeline.

3

Duplicate the Transformation View

In ADO, navigate to your development transformation view. Use the Duplicate option to create a copy. Rename the copy with a production label, e.g., Employee Transformation — Production.

This gives you a production transformation view that has the same transformation logic as Dev — same column casts, same filters, same joins. You just need to update the source in the next step.

Key Constraint

There is no "Clone to LLM models" feature for transformation views. Unlike links, transformation views cannot be promoted via a single click. You must manually duplicate the view within ADO and update its source reference. This is expected behavior — plan for it in your project timeline.

📸

4

Update the Transformation View Source to Production

Open the duplicated production transformation view. Change its source dataset from the development source data to the production source data you created in Step 2.

Verify that the column mappings still align — if your production CSV has the same column headers as dev (which it should), no remapping is needed. If column names differ, update the mappings at this step.

📸

5

Create the Production Link Using Clone to LLM Models

This is the step where ADO's one ALM-like feature comes in. Navigate to your development link in ADO. Click the Clone to LLM models option (sometimes shown as "Clone Link" or similar depending on the ADO version). A dialog will appear asking you to select the target model.

Select the production model you prepared in Step 1. ADO will create a new link that targets the production model and automatically creates the corresponding ADO import action in that model.

Critical Step After Cloning

Even after using Clone to LLM Models, the new production link still points at the development transformation view as its source. This is not changed automatically. You must manually update the production link's source to point to the production transformation view you created in Steps 3–4.

📸
[Screenshot Placeholder]

6

Duplicate the Workflow Template and Update

Return to Workflow. Find the development template you created in Lab A (e.g., ADO | Employee Data | Dev | Nightly). Duplicate it and rename it to the production equivalent: ADO | Employee Data | Prod | Nightly.

Inside the duplicated template, update the ADO Action step to reference the production link instead of the development link. Finally, set the production schedule (same cadence as Dev, or adjust if the customer wants production loads at a different time).

Expected Topology After Promotion

Before promotion, your ADO map looks like this (development only):

Development-only Topology

Dev Source Data → Employee TV (Dev) → Dev Link → HR Model (Dev)
Automated by: ADO | Employee Data | Dev | Nightly

After completing all 6 steps, your ADO map should look like this:

Full Dev + Prod Topology

Dev Source Data → Employee TV (Dev) → Dev Link → HR Model (Dev)
Automated by: ADO | Employee Data | Dev | Nightly

Prod Source Data → Employee TV (Prod) → Prod Link → HR Model (Prod)
Automated by: ADO | Employee Data | Prod | Nightly

ALM Deep Dive: What CAN vs. CANNOT Be Cloned

ADO Component Can Clone? How Post-Clone Action Required
Links ✅ Yes "Clone to LLM models" feature in ADO Must update source (transformation view) to production version
Transformation Views ⚠️ Manually only Use Duplicate within ADO; rename; update source Must update source dataset to production version
Source Data ❌ No Must create new production source dataset (upload CSV or create new connector) None after creation — this IS the production source
Workflow Templates ⚠️ Manually only Use Duplicate in Workflow; rename; update ADO Action step Update step to reference production link; reconfigure schedule
