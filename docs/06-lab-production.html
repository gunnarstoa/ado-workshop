Lab B: Promote Dev to Production

Walk the full 6-step production promotion for Meridian Corp's HR payroll ADO job
Lab B ⏱ 60 min Hands-On
Meridian Corp Scenario

Meridian Corp's Development phase is complete. The HR payroll ADO job is working in the Dev environment — employee data loads into the HR Model (Dev) via ADO nightly. The go-live date is approaching. Your task is to promote the HR payroll ADO pipeline to the Production environment so that live payroll data loads into HR Model (Prod) automatically.

What You'll Need

  • A working ADO pipeline in your Dev environment (HR Model ADO job from Academy course)
  • A production Anaplan model (or use a model you designate as "production" for this exercise)
  • A production CSV file — you can use the same sample CSV from the Academy course for this exercise, just rename it to indicate it's "production data"
  • Workspace Admin access in both the Dev and Prod workspaces (or the same workspace if using a single workspace)
1
Clone the Model to LLM (Create Production Model)
⏱ 10 min
Objective Create or confirm the existence of a production HR model. This is the Anaplan model that the production ADO link will push data into.
  1. 1
    Navigate to the Model Management area of your workspace (via the Workspace Admin menu).
  2. 2
    If you already have a production HR model from a previous ALM promotion: confirm it is up-to-date and note its name. You will select it in Mini-Lab 4.
  3. 3
    If you do not have a production model: duplicate the Dev HR model. Click the three-dot menu on the HR Model (Dev) and select Duplicate. Name it HR Model (Prod).
  4. 4
    Open the production model and verify it has the correct module structure (same lists, modules, and line items as Dev). Data can be empty — ADO will populate it.

2
Create Production Source Data (CSV Import)
⏱ 10 min
Objective Upload a production employee CSV file and create a new source dataset in ADO. This is the production data source that will feed the production pipeline.
  1. 1
    Navigate to ADO in your Anaplan workspace. Go to Source Data.
  2. 2
    Click Add Source Data → Select File Import.
  3. 3
    Upload your production employee CSV file. For this exercise, use the same CSV from the Academy course and rename it employee_data_prod.csv to distinguish it.
  4. 4
    Name the source dataset: Employee Data — Production.
  5. 5
    Confirm column headers are detected correctly (same as the Dev source data — same CSV structure).
  6. 6
    Save the source dataset.

✅ Done when:
  • "Employee Data — Production" appears in your ADO Source Data list
  • Columns are correctly detected and match the Dev source data structure
3
Duplicate Transformation View and Update Source
⏱ 10 min
Objective Create a production transformation view by duplicating the Dev transformation view and pointing it at the production source data.
  1. 1
    In ADO, go to Transformation Views.
  2. 2
    Find your development employee transformation view. Click the three-dot menu → Duplicate.
  3. 3
    Rename the duplicate to Employee Transformation — Production.
  4. 4
    Open the production transformation view. Find the Source Data setting at the top of the view configuration.
  5. 5
    Change the source from Employee Data — Dev to Employee Data — Production.
  6. 6
    Verify that all transformation steps (column casts, filters, joins) still appear correctly. If column names are identical between Dev and Prod source data, everything should resolve without changes.
  7. 7
    Save the production transformation view.
📸

✅ Done when:
  • "Employee Transformation — Production" exists in your Transformation Views list
  • Its source is "Employee Data — Production" (not Dev)
  • All transformation steps are intact and show no errors
4
Create Production Link Using Clone to LLM Models
⏱ 10 min
Objective Use ADO's "Clone to LLM models" feature to create a production link targeting the HR Model (Prod), then update its source to the production transformation view.
  1. 1
    In ADO, go to Links. Find your development HR link (the one that loads employee data into HR Model (Dev)).
  2. 2
    Click the three-dot menu on the development link → Select Clone to LLM models.
  3. 3
    A dialog appears asking you to select the target model. Select HR Model (Prod) from the dropdown.
  4. 4
    Click Clone. ADO creates a new link targeting the production model and automatically creates an ADO import action inside HR Model (Prod).
  5. 5
    Find the newly created production link in the Links list. It will have the same name as the Dev link but reference HR Model (Prod).
  6. 6
    Critical: Open the production link. Find the Source setting. Note that it currently points to the development transformation view — this was not changed automatically. Change it to Employee Transformation — Production.
  7. 7
    Save the production link.
Easy to Miss

The "Clone to LLM models" feature creates a production link pointing at the correct production model, but it does not update the source to the production transformation view. If you skip Step 6, production data will actually come from the development transformation view — meaning it's using dev source data. Always check and update the source after cloning.

5
Duplicate Workflow Template and Update Action
⏱ 10 min
Objective Create a production Workflow template by duplicating the Dev template and updating the ADO Action step to use the production link.
  1. 1
    Navigate to Workflow. Find the template ADO | Employee Data | Dev | Nightly.
  2. 2
    Click the three-dot menu → Duplicate.
  3. 3
    Rename the duplicate to ADO | Employee Data | Prod | Nightly.
  4. 4
    Open the production template. Find the ADO Action step that references the Dev link. Click to edit the step.
  5. 5
    Change the ADO link from the development link to the production link you created in Mini-Lab 4.
  6. 6
    Go to the Schedule tab. Confirm the schedule settings (Daily, 23:00 UTC). Adjust if the production environment has a different requirement.
  7. 7
    Enable the schedule and save.
✅ Done when:
  • "ADO | Employee Data | Prod | Nightly" exists in Workflow, scheduled and enabled
  • Its ADO Action step references the production link (not the Dev link)
6
Verify the Production Map
⏱ 10 min
Objective Run the production ADO link manually to confirm the full production pipeline works, then draw the expected ADO topology to verify your understanding.
  1. 1
    In Workflow, open ADO | Employee Data | Prod | Nightly. Click Run Now.
  2. 2
    Watch the Run History. Wait for "Completed" status.
  3. 3
    Navigate to HR Model (Prod). Open a module that should contain employee data. Verify data is present.
  4. 4
    On paper or in a drawing tool, draw the expected ADO topology. Your drawing should match the sample answer below.

Sample Answer: Expected ADO Topology After Promotion

Complete Production Topology — Meridian Corp HR Payroll

Development Pipeline:
Employee Data (Dev CSV) → Employee Transformation — Dev → Dev Link → HR Model (Dev)
Automated by: ADO | Employee Data | Dev | Nightly (23:00 UTC)

Production Pipeline:
Employee Data (Prod CSV) → Employee Transformation — Production → Prod Link → HR Model (Prod)
Automated by: ADO | Employee Data | Prod | Nightly (23:00 UTC)

Result: Two independent pipelines. Dev continues running for ongoing development work. Prod runs production data to the live model. No cross-contamination.

Congratulations

You've completed the full ADO delivery cycle — from development build (Academy course) through automation (Lab A) and into production (Lab B). This is the pattern you'll replicate on every customer ADO implementation.
