Q&A from Sessions

Real questions from ADO partner sessions, with full answers
Reference Session: March 2026

These Q&A pairs come from actual partner sessions and implementation discussions. They cover common questions about Workflow, ALM, prerequisites, and production patterns.

Workflow & Scheduling

Q: Is Workflow something that customers have to buy separately, or is it included in Anaplan?
A: In V5 pricing (which covers both the Advanced and Standard tiers), Workflow is included. Essentially all customers on V5 pricing can use Workflow — there is no separate purchase or add-on required. If a customer is on an older pricing structure, check with the Account Executive to confirm. For most partners working on active apps opportunities, Workflow availability is not a concern.
Q: Can I put multiple ADO jobs in a single Workflow template, or should I keep them separate?
A: Technically you can add multiple ADO Action steps to a single Workflow template, but it's a best practice to keep them separate — one template per dataset type. The reason: if you combine employee data and product data in one template, a failure in one dataset blocks or delays the other. Separate templates give each dataset its own run history, its own failure scope, and its own schedule. They're also easier to debug and promote independently to production.
Q: Workflow schedules are in UTC — how do I handle a customer that wants their data refreshed at 8 AM local time in Tokyo?
A: Japan Standard Time (JST) is UTC+9, with no daylight saving time. So 8 AM JST = 23:00 UTC the previous day. Set the Workflow schedule to 23:00 UTC daily, and the data will be ready in the Anaplan model by 8 AM the following morning in Tokyo. Always document the UTC schedule and its local-time equivalent in your project notes — it's a common source of confusion when handover documentation is unclear.
Q: Is there a minimum interval for Workflow schedules — can I schedule ADO to run every 5 minutes?
A: The minimum practical interval for Workflow-triggered ADO runs depends on the ADO pipeline duration. ADO runs block the Anaplan model during the load phase — running every 5 minutes would keep the model essentially permanently locked. Additionally, the Workflow scheduler has minimum interval constraints (typically 15 minutes). For near-real-time data needs (<15 minute intervals), consider a direct Anaplan API integration rather than Workflow-scheduled ADO. ADO is designed for batch-oriented scheduled loads.

ALM & Productionization

Q: Is there an ALM function in ADO — can I promote my ADO pipeline the same way I promote a model?
A: Yes and no. ADO has one ALM-like feature: the "Clone to LLM models" option on links, which creates a production-targeted link from a development link in one click. However, that's the extent of it. Transformation views and source data do not have an equivalent promote feature — they must be manually duplicated within ADO and their sources updated. This means productionizing ADO requires more manual steps than promoting a model. Build this into your project plan: allow 2–4 hours per ADO pipeline for productionization, depending on complexity.
Q: After using "Clone to LLM models" to create a production link, does ADO automatically update the source to the production transformation view?
A: No — and this is the most common mistake in ADO productionization. "Clone to LLM models" updates the target model of the link (pointing it at the production model), but it does NOT update the link's source. After cloning, the production link still points at the development transformation view, which reads from development source data. You must manually open the production link after cloning and update its source to the production transformation view. Forgetting this step means production loads are silently pulling dev data.
Q: Can I skip the Academy course and jump straight to this workshop if I have general Anaplan experience?
A: You can follow the steps in this workshop without completing the Academy course, but you'll be missing important context — specifically, the HR model and FPA model that Labs A and B are designed around. The Academy ADO course provides both the conceptual foundation (what ADO components are and how they connect) and the hands-on models you'll use in these exercises. Without those, you'll either need to build your own equivalent models first or adapt the labs significantly. We strongly recommend completing the Academy ADO course before starting this workshop.

Source Data & Transformation

Q: Should we use a connection or a CSV file for the ADO source data in production? What's the recommendation?
A: For production implementations, live connectors are generally preferred because they eliminate the manual step of uploading a file — data flows directly from the source system on schedule. However, many Academy exercises and smaller implementations use CSV because it's simpler to configure. The choice depends on the customer's technical setup: if they have a supported connector for their source system (Salesforce, SAP, Snowflake, Workday, etc.) and their IT team can configure the connection, use a connector. If the source system doesn't have a native ADO connector, or if the data volume and complexity are low, CSV-based imports via CloudWorks or API are perfectly valid for production.
Q: Can ADO read data from another Anaplan model — like pulling actuals from a data hub model into a planning model?
A: Technically yes, but it's generally not recommended for model-to-model transfers within Anaplan. ADO is optimized for bringing external data into Anaplan. For moving data between Anaplan models (like actuals from a data hub into a planning spoke), the native Anaplan import action (model-to-model import) is the right tool — it's faster, simpler, and doesn't consume ADO storage or pipeline capacity. Use ADO for what it's designed for: external data ingestion with transformation. If you do use ADO to source from an Anaplan model, follow Planual rule 8.01-01: create a dedicated flat module in the source model with only the data ADO needs — never source from a module with subsidiary views.
Q: In a customer project, how many ADO pipelines should I expect to need? Is it one pipeline for everything?
A: Plan for one pipeline (link + transformation view chain) per distinct dataset type. A typical mid-market implementation might have 3–5 pipelines: employee master, headcount actuals, product master, financial actuals, and a budget upload. Each dataset has its own source structure, transformation logic, and load schedule — they should not be combined. If two datasets have the same source structure and schedule, it may be worth combining them in one transformation view with a join, but in practice, separate pipelines per dataset are easier to maintain, debug, and promote to production independently.
